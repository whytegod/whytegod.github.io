Anchoring a Record

Anchoring is the act of committing a Record ID to an immutable public system. Once anchored, the record gains permanent verifiability.

What You Anchor

  • Only the hash (Record ID)
  • Never the original data
  • Privacy is preserved

Step-by-Step

  1. Generate a Record ID using the Hash Generator
  2. Choose an immutable public anchor:
    • Bitcoin OP_RETURN
    • Ethereum calldata
    • Public timestamping services
    • Public append-only logs
  3. Publish the Record ID as plain text or hex
  4. Save the transaction reference or anchor ID

What Makes It Valid

  • The hash is deterministic
  • The anchor is immutable
  • The timestamp is externally verifiable
  • No trust in Whytegod is required

Verification Later

To verify a record:

  1. Recompute the hash from the original data
  2. Compare it to the anchored Record ID
  3. Match confirms authenticity

Whytegod does not run nodes, store data, or control anchors. The protocol defines structure and verification — nothing more.

Recommended Anchor Formats

  • UTF-8 encoded hash string
  • Hexadecimal representation
  • Base64 (optional, must be documented)

The format used must remain consistent for future verification.

Anchor Metadata (Optional)

You may publish metadata alongside the Record ID:

  • Anchor date
  • Anchor network
  • Transaction or reference ID
  • Human-readable description

Metadata is not part of verification — the hash alone is sufficient.

Security Considerations

  • Never reuse hashes across different datasets
  • Do not publish unhashed sensitive data
  • Preserve original files exactly as hashed
  • Document anchor locations clearly

Common Use Cases

  • Legal documents & agreements
  • Research papers & datasets
  • Intellectual property proofs
  • Audit trails & compliance records
  • Personal archives

Anchoring is optional but recommended for records requiring strong timestamp guarantees and public verifiability.
