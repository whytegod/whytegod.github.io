Protocols

Protocols define how records are created, anchored, and verified. They are rules, not services.

Why a Protocol?

  • To avoid centralized trust
  • To ensure long-term verifiability
  • To separate proof from platforms
  • To survive software, website, and ownership changes

Core Protocols

1. Hashing Protocol

Defines how original data is transformed into a Record ID using deterministic cryptographic hashing.

  • Input: Any digital data
  • Output: Fixed-length hash
  • No data storage

2. Anchoring Protocol

Defines how Record IDs are committed to immutable public systems.

  • Blockchain transactions
  • Public append-only logs
  • Timestamping services

3. Verification Protocol

Defines how anyone can independently verify a record.

  • Recompute hash
  • Match Record ID
  • Confirm anchor existence

User Interaction Flow

  1. User generates a Record ID locally
  2. User anchors the Record ID publicly
  3. User stores anchor reference
  4. Anyone can verify without permission

What Whytegod Does NOT Do

  • Does not hold user data
  • Does not validate records
  • Does not reverse or edit records
  • Does not require accounts or identity

Extensibility

Any system may implement Whytegod protocols without permission. The protocol is compatible with:

  • Legal evidence systems
  • Academic publishing
  • Supply chain logs
  • Personal proof-of-existence

Protocols are stable by design. Upgrades add clarity — not dependency.
