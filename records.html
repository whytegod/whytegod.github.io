Records

A Record represents a claim, event, or state that can be proven to have existed at a specific time. Whytegod records do not store data — they define how proof is created.

What a Record Is

  • A cryptographic fingerprint of information
  • Generated deterministically
  • Independent of storage or platforms
  • Verifiable by anyone, anytime

What a Record Is NOT

  • Not a database entry
  • Not stored by Whytegod
  • Not reversible
  • Not editable

Record ID

Every record is represented by a Record ID, which is a cryptographic hash of the original data.

  • Same input → same Record ID
  • Different input → different Record ID
  • Data never leaves the owner

Record Lifecycle

  1. Create original data (document, event, file, statement)
  2. Generate a Record ID using hashing
  3. Anchor the Record ID publicly
  4. Verify later by recomputing the hash

Verification

A record is verified when:

  • The recomputed hash matches the Record ID
  • The Record ID exists in a public immutable anchor
  • The timestamp aligns with the claim

Trust Model

Whytegod does not act as a trusted third party. Verification depends only on cryptography and public systems.

Records remain valid even if the Whytegod website disappears. The protocol defines rules, not control.
