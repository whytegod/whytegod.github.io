Verification

Verification proves that a specific piece of data existed at or before a specific point in time. This process is deterministic, permissionless, and public.

What Verification Confirms

  • The data existed
  • The data has not changed
  • The data was anchored at a known time
  • The record belongs to the public ledger

What Verification Does Not Confirm

  • Who created the data
  • Why the data exists
  • Ownership or authorship
  • Legality or truthfulness

Verification Requirements

  • The original data (file, text, or dataset)
  • The hashing algorithm used (SHA-256)
  • Access to the public record list

Verification Process

  1. Reproduce the original data exactly
  2. Generate a SHA-256 hash from the data
  3. Locate the matching hash in the records list
  4. Confirm the timestamp and anchor order

Local Hashing

Hashing must always occur locally. The protocol never requires file uploads for verification.

  • Use offline tools
  • Use open-source hash utilities
  • Never trust third-party conversions

Matching Rules

A verification match is valid only if:

  • The hash matches exactly (character-for-character)
  • The hashing algorithm matches
  • The record appears in the ordered ledger

Time Meaning

The timestamp indicates the moment the hash was anchored — not when the data was created.

Creation may have occurred earlier. Anchoring defines the latest possible existence.

Public Verifiability

Verification does not require accounts, tokens, permissions, or trust in Whytegod itself.

Anyone can verify using independent tools.

Dispute Handling

The protocol does not resolve disputes. Verification results are factual outputs.

Interpretation is external to the system.

Failure Scenarios

  • If the data changes → verification fails
  • If the hash differs → verification fails
  • If the record is absent → verification fails

Verification answers only one question: Did this exact data exist by this time?
